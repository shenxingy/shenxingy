# Design System — `shenxingy/shenxingy`

> Scope: the GitHub profile README. This file is the single source of truth for all visual decisions on the profile. Any change to the README's look must be reflected here first.

**Aesthetic**: Extreme minimal. One typographic move — a serif name — on an otherwise stock-markdown page.
**Differentiation** (the one thing to remember): the serif name-mark. It is the only departure from GitHub's default sans rendering, and therefore it has to carry the whole page.
**Viewport constraint**: the full README must fit within a single laptop viewport (≤ ~700px total rendered height). Every spacing and sizing decision is subordinate to this.
**What we explicitly gave up in v2**: editorial tropes (VOL · I, MMXXVI Roman-numeral year stamp, § section numerals, hairline rules, accent-color endcap). They were decoration that required the viewer to decode them — if a reader has to ask "what does this mean?", the signal didn't reach them.

---

## 1. Rendering Constraints (read first)

GitHub renders profile READMEs through a sanitized markdown pipeline. Every token below is chosen to survive it.

| Allowed                                           | Stripped / unusable          |
|---------------------------------------------------|------------------------------|
| `<a>` `<img>` `<picture>` `<div align=>` `<br>` `<hr>` | `<svg>` (inline — must be external file) |
| `<samp>` `<sub>` `<sup>` `<kbd>` `<code>`          | `<style>` `<script>`         |
| `<table>` `<details>` `<summary>`                 | Custom CSS, class attrs      |
| External SVGs via `<img src="assets/*.svg">`      | Web fonts on the body        |
| `<picture>` with `prefers-color-scheme` media     | JavaScript of any kind       |

**Implication**: custom typography is only possible inside `.svg` files. Body text uses GitHub's default system font stack — we do not fight it.

---

## 2. Typography

Two surfaces: the SVG name-mark (serif) and everything else (GitHub default).

### Display (inside the masthead SVG)

- **Primary serif** — `font-family="ui-serif, 'Iowan Old Style', 'Apple Garamond', Baskerville, 'Times New Roman', Georgia, serif"`
  - High-contrast serif fallback chain. Lands on Iowan on macOS, Cambria/Georgia on Windows, DejaVu Serif on Linux.

### Body (markdown default)

- GitHub's sanitized default (`-apple-system, BlinkMacSystemFont, 'Segoe UI', ...`). Do not attempt to override.

### Type scale

| Token          | Size | Weight | Use                                         |
|----------------|------|--------|---------------------------------------------|
| `display-name` | 32   | 400    | Masthead name "Alex Shen" (the only use)    |

No other typographic tokens. Body text inherits GitHub's rendering at whatever size the viewer's device provides.

### Case & tracking

- Name-mark: **sentence case** ("Alex Shen"), roman (not italic), no tracking. Calmer than tracked all-caps; less "LOOK AT ME".
- Italic is reserved for at most one phrase in the body (e.g. a one-line tagline). Not currently used in v2.

## 3. Color

Pure monochrome. Ink on transparent — the SVG shows GitHub's page color through, and the text color is what we control.

### Tokens

| Token          | Light mode | Dark mode | Use                          |
|----------------|------------|-----------|------------------------------|
| `--ink`        | `#121212`  | `#ECE8E0` | Name-mark fill               |
| `--ink-muted`  | `#5A5A5A`  | `#9A9690` | Reserved — not in v2 README  |
| `--accent`     | `#B8301C`  | `#D9533F` | Reserved — not in v2 README  |

**Rules**:
- No accent color on the page. Not a single pixel. Monochrome is the whole point.
- SVG backgrounds are **transparent** (no `<rect fill=>` behind the text). This removes the need to chase GitHub's exact pane color and lets the image blend in automatically.
- Text color is the only thing that differs between light and dark variants.

### Dark-mode delivery pattern

```html
<picture>
  <source media="(prefers-color-scheme: dark)" srcset="assets/masthead-dark.svg">
  <img src="assets/masthead-light.svg" alt="Alex Shen" width="680">
</picture>
```

## 4. Spacing & Rhythm

Default markdown rhythm + a few explicit `<br>` breaks. The README should read like a short text document, not a magazine spread.

### Vertical rhythm (top-to-bottom)

| Step | Mechanism      | Use                                     |
|------|----------------|-----------------------------------------|
| S    | blank line     | Paragraph break between project rows    |
| M    | single `<br>`  | Between the masthead/tagline and the project block; between the project block and the contact footer |

No `<br><br>`. No hairline rules. No horizontal dividers.

### Horizontal alignment

- Entire README stays in `<div align="center">`. Centered is the right feel for a profile card (not a documentation page).

### Inline spacing

- Between inline items on one line: `&nbsp;·&nbsp;` (middle dot with non-breaking spaces, so phrase boundaries don't wrap awkwardly).
- Em dashes (`—`, U+2014) between project name and description. Never hyphen-minus.
- No emoji. Anywhere.

## 5. Components

Two components. That is the entire system.

### 5.1 Masthead `assets/masthead-{light,dark}.svg`

Just the name. Nothing else.

```
┌──────────────────────────────────────────────────┐
│                                                  │
│                   Alex Shen                      │
│                                                  │
└──────────────────────────────────────────────────┘
```

- Canvas: `viewBox="0 0 680 56"`. Rendered at `width="680"`.
- Text: "Alex Shen" at 32px serif, weight 400, `fill="--ink"`, centered (`text-anchor="middle"` at x=340, baseline y=38).
- Background: transparent. No `<rect>`, no border, no decoration.
- Light variant fills `#121212`; dark variant fills `#ECE8E0`. Everything else is identical.

### 5.2 Project Row (markdown)

```markdown
[project-name](URL) — one-line description &nbsp;·&nbsp; [source](SOURCE)
```

- Project name lowercase (matches the repos' actual names), as a markdown link.
- Em-dash separator.
- Description is verb-led, sentence case, no marketing adjectives.
- Optional `· source` tail only when a project has both a deploy URL and a source link.

## 6. Components that were in v1 but are removed in v2

Keep this list. It is load-bearing — it explains the design's direction.

| Removed         | Why                                                         |
|-----------------|-------------------------------------------------------------|
| `VOL · I`       | Required the viewer to know "volume 1 = first issue of this profile-as-publication". Not obvious; not worth the friction. |
| `MMXXVI`        | Roman numerals are a trope, not a signal. Nobody needs the year stamped on a GitHub profile. |
| Hairline rule + accent endcap in masthead | Decoration without information. Monochrome without the endcap is cleaner. |
| `§ 01 / § 02 / § 03` section numerals | Borrowed from magazine layout. Doesn't add clarity to a 5-project list; adds visual weight. |
| Inter-section hairline rules | Same — the list has clear rhythm from paragraph breaks; rules were belt-and-suspenders. |
| Subtitle line inside the SVG | Duplicated the role info that the body's tagline already carries; made the masthead taller without new info. |

`assets/rule-light.svg` and `assets/rule-dark.svg` remain in the repo as reserved components, available for future surfaces (a blog header, a long-form section break). They are **not** referenced by the v2 README.

## 7. Voice & Copy Rules

The visual system sets a tone of restraint; copy must match.

- **Verb-led project descriptions**. "building X" not "a system that helps users with X".
- **No adjectives of self-evaluation**: `passionate`, `experienced`, `skilled`, `innovative` — all banned.
- **Sentence case** for descriptions. No tracked all-caps anywhere.
- **Em-dashes, not hyphens**, between name and description.
- **No emoji**. Anywhere.
- **Links use the project's canonical name**. Not "here", not "my site".
- The tagline, if kept, is **one sentence**. Currently: *Founding Engineer at Scam AI. Building tools to detect AI-powered fraud.*

## 9. Verification Rules (mirrored in `VERIFY.md`)

Checkpoints any README change must pass before push:

1. All `<a href>` URLs resolve (2xx/3xx; LinkedIn's 999 anti-bot response counts as resolve).
2. All referenced SVG assets exist at `assets/*.svg` and are valid XML.
3. Every `<picture>` has both a default `light` and a `prefers-color-scheme: dark` source.
4. README line count stays within 20–50 lines (business-card scope).
5. Rendered-height budget ≤ 700px (masthead ≤ 56px; no `<br><br>`; no hairline rules in README).
6. No accent color (`#B8301C` / `#D9533F`) appears anywhere — monochrome is a hard rule in v2.
7. No raw `<svg>` / `<style>` / `<script>` tags in README (sanitizer-stripped; catch at author time).

---

## 10. Out of Scope

- Dynamic GitHub stats / contribution graphs.
- Third-party image services (shields.io, etc.).
- Animated SVGs.
- Section kickers, numerals, rules, accent colors — removed in v2 (see §6) and not to be reintroduced without an explicit aesthetic decision that replaces extreme-minimal with something else.
